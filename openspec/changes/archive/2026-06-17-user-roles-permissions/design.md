## Context

Roles already exist (`IdentityRole` via `AddRoles<IdentityRole>()`), are seeded in `Program.cs` (`SuperAdmin`, `Admin`), and are already embedded in the JWT as `ClaimTypes.Role` claims (`AuthService.IssueTokensAsync`). Nothing currently checks them — no `[Authorize(Roles=...)]`, no policies, no permission concept. `ApplicationUser` has exactly one extra field beyond `IdentityUser` (`DisplayName`).

Research of InventoryMSNV's equivalent (`SystemRoleController`, `ManageUserRolesController` + `ManageUserRoles`/`ManageUserRolesDetails`, `UserManagementController`) found a confusing three-tier structure: raw Identity roles → a separate "role profile" abstraction (a checkbox matrix of ~80 page-level Identity roles per profile) → users picking one profile from a dropdown, which then materializes the profile's checked pages back into real Identity role memberships on save. The actual authorization at runtime is still just page-level `[Authorize(Roles=...)]` per controller — coarse, not action-level, and the "role profile" layer exists only to make curating which pages a role can see less tedious by hand.

LitXusCount does not need that complexity: ASP.NET Core Identity already supports per-role claims (`AspNetRoleClaims`) out of the box, which is exactly a permission-assignment mechanism with no new schema required.

## Goals / Non-Goals

**Goals:**
- Real user CRUD (list/create/edit/deactivate/admin-reset-password) replacing the two hardcoded dev accounts as the only way in.
- Real role CRUD with a genuine action-level permission catalog (View/Create/Edit/Delete per resource) assigned via role claims — not page-level, not a separate "role profile" abstraction.
- A dynamic permission policy provider so `[Authorize(Policy = "Permission.X")]` works without hand-registering every permission as a policy.
- `SuperAdmin` protected from deletion and from ever losing effective access, even if its claims are misconfigured.
- **Retrofit the same per-entity permission model onto the 9 already-built System Settings endpoints and pages**, establishing the pattern every future module is expected to reuse from day one, rather than treating System Settings as a permanent `[Authorize]`-only exception.

**Non-Goals:**
- No per-user permission overrides — permissions are assigned to roles only; a user's effective permissions are the union of their roles' permissions. Simpler, and matches how every other part of this codebase favors the plainest mechanism that works (e.g. no MediatR/CQRS, no generic repository).
- No org-chart/branch/department scoping on users (InventoryMSNV's `UserProfile.BranchId`) — LitXusCount has no branch/multi-location concept yet; out of scope until it does.
- No self-service "my profile" page — this change is the *admin* managing users/roles, not a logged-in user editing their own profile. Could be a later, much smaller change.
- No hard delete for users — see Decision 2.

## Decisions

1. **Permissions are role claims (`AspNetRoleClaims`), not a new Domain entity/table.** Each permission is a claim with type `"permission"` and value equal to a permission string (e.g. `"Users.Edit"`). `RoleManager<IdentityRole>.AddClaimAsync`/`RemoveClaimAsync` manage them; `RoleManager.GetClaimsAsync(role)` reads them back for the role-edit screen's checkbox matrix. Alternative considered: a custom `Permission`/`RolePermission` table — rejected, since it would duplicate what `AspNetRoleClaims` already does, adding a migration and two new entities for no behavioral gain.

2. **The permission catalog itself is a compile-time C# constants class** (`Permissions.Users.View`, `Permissions.Users.Create`, etc. in `LitXusCount.Application`), not a database table. The set of *available* permissions is part of the application's code (new permissions ship with new code, like routes), while which permissions a *role* has is the only thing that's actually dynamic/admin-configurable. The role-edit screen's checkbox matrix is generated by reflecting over this catalog. Alternative considered: a `Permissions` DB table seeded from code — rejected as unnecessary ceremony; nothing needs to query "all permissions" except this one screen, which can just enumerate the C# catalog directly.

   The catalog is **per-entity, not one umbrella permission per module** — System Settings gets its own nested class per entity (`Permissions.Settings.Currency.{View,Create,Edit,Delete}`, `Permissions.Settings.Category.*`, etc., 8 entities × 4, plus `Permissions.Settings.CompanyInfo.{View,Edit}` since that one has no create/delete), confirmed directly over the simpler alternative (one `Settings.*` set covering all 9 entities) because the user wants to be able to grant, say, "view Currency but not Category" to a single role. This makes the catalog 2 levels deep in places (`Settings.Currency.Delete`) rather than 1 (`Users.Delete`); the enumeration helper that walks the catalog for the role-edit checkbox matrix and for seeding therefore recurses through nested classes rather than assuming a single level.

3. **Dynamic permission policy provider** (`PermissionAuthorizationHandler` + a custom `IAuthorizationPolicyProvider`) lets controllers write `[Authorize(Policy = "Permission.Users.Edit")]` directly, with the policy created on-demand the first time it's requested (checking `User.HasClaim("permission", "Users.Edit")`), rather than requiring every permission to be pre-registered via `AddPolicy` in `Program.cs`. Standard, well-known ASP.NET Core pattern for claims/permission-based authorization. Alternative considered: requiring every permission to be registered as a named policy up front — rejected, since the list would need editing every time a permission is added, which defeats the point of a compile-time catalog.

4. **JWT must carry the user's *effective* permission claims, aggregated across all their roles, not just role names.** `IssueTokensAsync` already adds `ClaimTypes.Role` per role; it now additionally calls `RoleManager.GetClaimsAsync(role)` for each role the user belongs to and adds each `"permission"` claim found, de-duplicated. This keeps the permission check (`User.HasClaim`) a pure JWT-claims check with no extra DB round-trip per request — consistent with the existing "JWT bearer only" decision in `identity-jwt-auth`.

5. **`SuperAdmin` is special-cased in code, not via a DB flag.** `RoleManagementService` rejects any delete/edit-permissions request where the role name is `"SuperAdmin"` (case-insensitive), and `PermissionAuthorizationHandler` short-circuits to "succeed" whenever the caller has the `SuperAdmin` role, regardless of which permission claims actually exist on that role. Alternative considered: an `IsBuiltIn` column on a custom role entity — rejected per Decision 1 (no new role table); a hardcoded name check is sufficient since `SuperAdmin` is a fixed, known constant seeded by this same codebase.

6. **User deactivation reuses Identity's existing lockout mechanism** (`LockoutEnd = DateTimeOffset.MaxValue`, `LockoutEnabled = true`) rather than adding a new `IsActive`/`IsDeleted` column to `ApplicationUser`. "Active users" list filters out anyone currently locked out indefinitely; "deactivate" sets it, "reactivate" clears it (`LockoutEnd = null`). Alternative considered: a new boolean column — rejected, since Identity already has a first-class mechanism for "this account cannot log in" that the login flow (`AuthService.LoginAsync`) already checks (`IsLockedOutAsync`).

7. **No hard delete for users.** Once other modules exist, rows will reference `CreatedBy`/`ModifiedBy` (string user IDs, per the audit-field convention already used in `AuditableEntity`) — hard-deleting a user would either dangle those references or require cascading deletes through unrelated modules. Deactivation (Decision 6) achieves the actual goal ("this person can no longer log in") without that risk.

8. **`LookupControllerBase<TService>` (the shared base class behind 5 of the 9 System Settings controllers — PaymentType, PaymentStatus, CustomerType, Category, UnitOfMeasure) uses imperative authorization, not declarative `[Authorize(Policy=...)]`, on its shared method bodies.** A static attribute on a base-class method applies identically to every subclass; since each subclass needs a *different* resource name in its policy string (`Settings.PaymentType.Delete` vs `Settings.Category.Delete`), the base class instead exposes `protected abstract string ResourceName { get; }` (overridden per subclass, e.g. `"PaymentType"`), injects `IAuthorizationService`, and calls `await authorizationService.AuthorizeAsync(User, $"Permission.Settings.{ResourceName}.{Action}")` inside each method, returning `Forbid()` on failure. Alternative considered: drop the shared base class and give each of the 5 controllers its own fully-written actions with static attributes — rejected, since that reverts the exact duplication `LookupControllerBase` was introduced to avoid in `system-settings`. The 4 hand-written controllers (`CompanyInfoController`, `CurrenciesController`, `VatPercentagesController`, `EmailConfigsController`) don't have this problem and keep plain declarative `[Authorize(Policy=...)]` attributes per action.

9. **Frontend permission checks decode the JWT manually, with no new npm dependency.** A ~10-line helper (`api/jwt.ts`) splits the token on `.`, base64url-decodes the payload segment, and `JSON.parse`s it — sufficient for reading the `"permission"` claims already embedded by Decision 4, without adding a `jwt-decode` package for what's a few lines of code. A `usePermissions()` hook wraps this and exposes `hasPermission(name)`.

10. **The frontend hides controls/nav entries a user lacks permission for, rather than disabling them.** Confirmed directly — a hidden "Delete" button needs no explanation; a visibly disabled one invites "why can't I click this?" support questions for a permission model the end user may not need to understand. Applies uniformly: System Settings page buttons, System Settings sidebar sub-entries, and the new Users/Roles pages themselves.

## Risks / Trade-offs

- **[Risk] A misconfigured policy name (e.g. `"Permission.Users.Edti"`, a typo) silently falls through to "no claim found" → 403, rather than failing at compile time.** → Mitigation: the permission catalog constants (Decision 2) are the only sanctioned source of policy-name strings; controllers reference `Permissions.Users.Edit` (a C# constant) when building the policy string, not a hand-typed literal, so a typo is a compile error, not a runtime surprise.
- **[Risk] Aggregating permission claims at login time (Decision 4) means a permission change to a role doesn't take effect for already-logged-in users until their token expires/refreshes.** → Mitigation: acceptable — this is the same trade-off the existing JWT-only auth design already made for role changes (changing a user's role today already doesn't affect their current token either); access tokens are short-lived (`AccessTokenMinutes` in `JwtSettings`) by design.
- **[Trade-off] Per-entity View/Create/Edit/Delete across Users, Roles, and all 9 System Settings entities is 42 permissions, not 8** — a real checkbox matrix on the role-edit screen, grouped by resource so it stays scannable. Accepted per explicit direction (confirmed: per-entity, not one umbrella `Settings.*` set) — the whole point of building a permission system instead of reusing role names directly is being able to grant exactly "view Currency but not Category."

## Migration Plan

No new tables — `AspNetRoleClaims` already exists from the original Identity migration (`AddRoles<IdentityRole>()` was already configured in the foundation change). A small data step (not a schema migration) seeds the `SuperAdmin` role with every catalog permission claim and the `Admin` role with a reasonable starter subset (`Users.View`, `Roles.View`, plus full View/Create/Edit/Delete on every System Settings entity, since "Admin" is the company-admin persona expected to manage settings day-to-day) at the same dev-seed point in `Program.cs` where roles are already created.

## Open Questions

None — the permission mechanism (role claims), the catalog shape (compile-time constants), and the `SuperAdmin`/deactivate/no-hard-delete decisions were confirmed directly before writing this design.
